# Scheduler DST Bug Fix Summary

## Problem
The meeting reminder application was sending reminders at incorrect times for meetings that fall between 2am-3am local time (Europe/Berlin), with exactly 1 hour
 offset. This only affected meetings during DST transitions.

## Root Cause
In `app.py`, the `convert_to_berlin()` function had a critical bug:

```python
# BUGGY CODE (before fix):
def convert_to_berlin(utc_time_str):
    utc_dt = datetime.strptime(utc_time_str, '%Y-%m-%dT%H:%M:%S')
    berlin_dt = BERLIN_TZ.localize(utc_dt)  # WRONG: treats UTC time as Berlin time
    return berlin_dt
```

The bug: `BERLIN_TZ.localize()` was being called on a naive datetime that represented UTC time. This incorrectly treated the UTC time as if it were already Berl
in time.

## The Fix

```python
# FIXED CODE:
def convert_to_berlin(utc_time_str):
    utc_dt = datetime.strptime(utc_time_str, '%Y-%m-%dT%H:%M:%S')
    utc_dt_aware = UTC_TZ.localize(utc_dt)  # First: make it timezone-aware as UTC
    berlin_dt = utc_dt_aware.astimezone(BERLIN_TZ)  # Then: convert to Berlin
    return berlin_dt
```

The correct approach:
1. Parse the UTC time string into a naive datetime
2. Localize it as UTC using `UTC_TZ.localize()`
3. Convert to Berlin timezone using `astimezone(BERLIN_TZ)`

- OLD (buggy): 2024-03-31 01:30:00 CET+0100 ❌

March 31, 2024 is when Europe/Berlin transitions from CET to CEST (springs forward at 2am to 3am). The hour from 2am-3am doesn't exist on this date.
