# Timezone Bug Fix Summary

## Problem
The Python scheduler app was sending meeting reminders at incorrect times for meetings between 2am-3am local time (Europe/Berlin), with exactly 1 hour offset.

## Root Cause
In the `convert_to_berlin()` function in `/home/user/scheduler/app.py`, the code was incorrectly using:
```python
berlin_dt = BERLIN_TZ.localize(utc_dt)
```

This treats the parsed UTC datetime as if it were already in Berlin timezone, which is incorrect.

## The Bug's Impact
- For the "Maintenance Window Prep" meeting at 2024-03-31T01:30:00 UTC
- Buggy code converted it to: 01:30:00 CET (treating UTC time as Berlin time)
- Should have been: 03:30:00 CEST (correctly converted, accounting for DST)
- Result: 2-hour error (appears as 1-hour reminder time difference)

## The Fix
Changed the conversion to properly handle timezone conversion:
```python
utc_dt_aware = UTC_TZ.localize(utc_dt)  # First, make it timezone-aware as UTC
berlin_dt = utc_dt_aware.astimezone(BERLIN_TZ)  # Then convert to Berlin time
```

## Why It Only Affected 2am-3am Meetings
March 31, 2024 is when Europe/Berlin transitions from CET (UTC+1) to CEST (UTC+2).
At 2:00am local time, clocks "spring forward" to 3:00am.
Meetings in this window exposed the bug because:
- The incorrect localize() used CET offset (+1)
- The correct conversion accounts for the DST transition to CEST (+2)
- This created exactly 1 hour difference in reminder times

## Testing
Verified the fix handles DST transitions correctly:
- Old method: 2024-03-31T01:30:00 UTC → 01:30:00 CET (WRONG)
- New method: 2024-03-31T01:30:00 UTC → 03:30:00 CEST (CORRECT)

## Files Modified
- `/home/user/scheduler/app.py` - Fixed the `convert_to_berlin()` function
- Backup created at `/home/user/scheduler/app.py.bak`

```

```
